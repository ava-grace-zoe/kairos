# rtf 执行规范

目标：通过「审查发现问题 → 测试确认问题 → 自动修复 → 回归验证」闭环，确保每个修复都有测试证据支撑，避免误修和引入新问题。

设计文档：`docs/rtf/design.md`

## 审查对象

| 模式         | 触发方式                         | 输入                                |
| ------------ | -------------------------------- | ----------------------------------- |
| **代码变更** | 用户提供 PR / diff / commit 范围 | `git diff` 输出、PR URL、commit SHA |
| **功能模块** | 用户指定文件或目录               | 文件路径、目录路径、glob 模式       |

用户未明确指定时，默认使用 `git diff HEAD` 获取未提交的变更。无变更则要求用户指定范围。

## 一、探查阶段

### 1.1 确定审查范围

**代码变更模式**：

1. 获取 diff：`git diff`（未提交）、`git diff <base>...<head>`（commit 范围）、`gh pr diff`（PR）
2. 从 diff 中提取变更文件列表和变更行范围
3. 排除不需审查的文件（lock 文件、生成文件、二进制文件）

**功能模块模式**：

1. 扫描用户指定的文件/目录
2. 识别模块边界：入口文件、导出接口、内部依赖

产出：`reviewScope`——文件列表 + 每个文件的变更/关注行范围。

### 1.2 探查测试基础设施

检测项目的测试能力：

1. 测试框架：检查 `package.json`（vitest / jest）、`pyproject.toml`（pytest）、`Cargo.toml`（cargo test）等
2. 测试命令：从 `scripts` 字段或配置文件推导
3. 已有测试文件：按约定路径扫描（`__tests__/`、`*.test.*`、`*.spec.*`、`tests/`）
4. 与审查范围相关的已有测试：匹配文件名或 import 关系

产出：`testInfra`——框架名称、测试命令、已有相关测试文件列表。无测试框架时标记 `framework: null`。

## 二、审查阶段

对 `reviewScope` 内的代码进行审查，发现潜在问题。

### 审查方式

- 不预设固定检查清单，根据代码性质自适应推导关注维度
- 代码变更模式：重点审查变更部分，同时关注变更对上下文的影响
- 功能模块模式：全面审查模块的正确性和健壮性

### 问题结构

每个发现的问题产出：

```
id: issue-NN
severity: critical | warning | info
file: <文件路径>
line: <行号或行范围>
description: <问题描述>
evidence: <代码证据——引用具体代码片段>
reproduction_hint: <复现思路——供测试设计使用>
```

### 排序与过滤

- 按 severity 排序：critical → warning → info
- `info` 级别只记录，不进入测试-修复循环
- `critical` 和 `warning` 进入第三阶段

### 无问题快速路径

审查未发现 critical/warning 级别问题 → 直接输出审查通过报告（含 info 级别的观察），流程结束。

## 三、测试-修复循环

对每个 critical/warning 级别的问题，按优先级依次执行以下子循环。

### 3.1 测试设计

根据问题的 `description` 和 `reproduction_hint`，设计最小化测试用例：

- **有测试框架**（`testInfra.framework != null`）：生成可执行的测试代码文件
  - 文件位置：遵循项目已有测试的组织方式
  - 测试命名：`rtf-issue-NN` 前缀，便于识别
- **无测试框架**：生成可在 shell 中执行的验证脚本
  - 脚本以非零退出码表示问题存在，零表示问题不存在

测试目标：在当前代码下**应当失败**（确认问题真实存在）。

### 3.2 测试执行

运行测试，判断结果：

| 测试结果     | 含义           | 动作                                                              |
| ------------ | -------------- | ----------------------------------------------------------------- |
| **失败**     | 问题已复现     | 进入 3.3 修复                                                     |
| **通过**     | 问题未复现     | 标记 `false_positive`，跳到下一个问题                             |
| **执行错误** | 测试本身有问题 | 修正测试代码，重试一次；仍失败则标记 `test_error`，跳到下一个问题 |

### 3.3 自动修复

执行最小化修复：

1. 只修改与问题直接相关的代码
2. 修复后运行该问题的测试，期望**通过**
3. 修复后测试仍失败 → 分析失败原因，调整修复方案，重试
4. 重试上限：**2 次**（含首次共 3 次尝试）
5. 超限 → 回滚所有修改，标记 `fix_failed`，跳到下一个问题

回滚方式：`git checkout -- <modified_files>`（仅回滚本问题修改的文件）。

### 3.4 回归验证

修复测试通过后，验证不引入新问题：

1. 运行 `testInfra` 中识别的相关已有测试
2. 无已有测试 → 跳过回归，直接标记 `fixed`

| 回归结果 | 动作                                                |
| -------- | --------------------------------------------------- |
| **通过** | 标记 `fixed`，进入下一个问题                        |
| **失败** | 回滚修复，标记 `regression`，记录回归失败的测试名称 |

### 循环控制

- 每个问题独立走完「测试 → 修复 → 回归」，不批量修复
- 前一个问题的修复可能影响后续问题的复现，每个问题的测试在当前代码状态下执行
- 如果某个问题的修复导致后续问题不再复现，后续问题自然被标记为 `false_positive`

## 四、终止与报告

所有 critical/warning 问题处理完毕后，在对话中输出综合报告：

### 1. 审查摘要

- 审查模式（代码变更 / 功能模块）
- 审查范围：文件数量、变更行数
- 发现问题：按 severity 统计

### 2. 问题清单

每个问题的最终状态：

| 状态             | 含义                     |
| ---------------- | ------------------------ |
| `fixed`          | 已修复且回归通过         |
| `false_positive` | 测试未复现，非真实问题   |
| `fix_failed`     | 修复尝试失败，需人工处理 |
| `regression`     | 修复导致回归，已回滚     |
| `test_error`     | 无法构造有效测试         |
| `info_only`      | 仅供参考的观察           |

### 3. 变更摘要

- 所有 `fixed` 问题产生的代码变更 diff
- 新增的测试文件列表

### 4. 未解决项

- `fix_failed` 和 `regression` 的问题：附带失败原因和建议的修复方向
- `test_error` 的问题：附带测试设计的困难点
