description: 将当前工作进度保存到当前目录的 `.work-progress/` 目录，强恢复导向。
---

将当前工作进度保存到 `.work-progress/` 目录。核心目标：让下次恢复任务的 codeAgent **跳过"重新探索代码库"这个最贵的环节，直接开干**。

## 存储位置

约定为**命令执行所在目录**下的 `.work-progress/`。目录不存在则先创建。

> 不要写到 `docs/` 下——进度是给下次 AI 会话恢复用的工作记忆，与给人读的正式文档分离。

## 文件命名规范

`{标题}_{YYYYMMDDHHmm}.md`

- **标题**: 任务标题，连字符连接，简洁明了（如 `tool-approval-migration`）。
- **时间**: 文件创建时的时间戳，格式 `YYYYMMDDHHmm`，精确到分钟。

示例：`tool-approval-migration_202602121430.md`

## 状态管理

状态由目录位置决定，不在文件名中维护：

- `.work-progress/` → 未完成（进行中 / 暂停 / 暂时卡住都放这里）
- `.work-progress/archive/` → 已完成

> 不单独维护额外状态目录。卡住的信息写进文件里的「当前卡点 / 风险」即可，避免恢复时多一层状态分流。

## 执行步骤

1. **扫描目录** — 列出 `.work-progress/` 下的 `.md` 文件（不含子目录），了解已有未完成任务。除非用户明确要求查看历史，否则不扫描 `archive/`。

2. **判断是新建还是更新**：
   - **文件名优先**：如果用户通过 $ARGUMENTS 指定了完整文件名，则只更新该文件。
   - **关键词次优先**：如果用户指定了关键词，则只在匹配该关键词的未完成任务中选择；有明确匹配则更新，没有则新建。
   - **自动生成最后**：如果用户没有指定文件名或关键词，则从当前任务生成标题，先查找相关未完成任务；有明确相关文件则优先提议更新，否则新建。

3. **从对话中提取信息**，按以下模板生成/更新文件内容。
   提取时优先回答："如果换一个全新会话的 Agent 来接手，它需要知道什么才能不靠重新搜索就继续干？"

```markdown
# {任务标题}

> 状态: {进行中/暂停/卡住/完成} | 作用域: {涉及的子包或模块，如 services/xxx；无则填 全局}
> 创建: {YYYY-MM-DD HH:mm} | 更新: {YYYY-MM-DD HH:mm}

## 目标

{一句话描述要解决的问题，以及"做到什么程度算完成"的判定标准}

## 执行计划

{把任务拆成几个大步骤（阶段），每个步骤下挂子项 TODO，标注步骤状态。
恢复时一眼看清整体推进到哪、下一个该攻哪个步骤。}

### 1. {阶段名} — 已完成
- [x] {子项，附关键结论}
- [x] {子项}

### 2. {阶段名} — 进行中
- [x] {子项}
- [ ] {子项：当前改到哪、卡在哪、半成品散落在哪些文件}

### 3. {阶段名} — 未开始
- [ ] {子项}

## 下一步（恢复后第一件事）

{不要只写一行动作。写"动作 + 做这步必须先知道的前置信息"：
- **动作**：具体改哪里（带 file:line）或跑什么命令
- **前置约束**：做这步必须握住的关键点（如"改字段要同步 schema.ts 和 client.ts，否则类型不通""命名沿用 userId"），让接手者不必重新推导}

## 代码认知资产（免重新探索）

{本次会话"探索代码库的结晶"——关于**代码库本身**的结构性认知，不因本次任务而改变。
让下次 Agent 零成本复用，不必重新 grep / 读文件。

- **代码地图**：关键文件/函数负责什么、调用链、数据流向
  - `path:line` — {职责 / 在链路中的位置}
- **项目约定 / 机制**：不读半天代码看不出来的特有规律
  - 如"timezone 统一在 adapter 层转换""所有 X 都走 Y 中间件"
- **关键事实**：入口在哪、配置在哪、关键依赖关系等一手信息}

## 关键发现 / 决策

{**因本次任务而产生**的判断——技术结论、实验结果、方案选型理由、排除了哪些方案。
与"代码认知资产"的区分：认知资产是"代码库本来就是这样的"，本区块是"我们在这个任务里决定这样做的"。}

## 死路 / 坑（别重复踩）

{试过但走不通的路径、踩过的坑，附简要原因。没有则写"暂无"}

## 验证方式

{怎么验证这个任务：跑哪个测试、用什么 probe、复现命令。写具体可复制的命令}

## 当前卡点 / 风险

{没有则写"暂无"。如果当前卡住，写清：
- **卡在什么上**：具体问题
- **依赖什么才能解除**：等谁/等什么条件/需要验证什么
- **解除后从哪里继续**：指向执行计划中的具体步骤}
```

4. **状态变更**：
   - 任务完成 → 将文件移动到 `.work-progress/archive/`。
   - 任务未完成、暂停或暂时卡住 → 文件留在 `.work-progress/`，并按需填写「当前卡点 / 风险」。
   - 从归档恢复继续做 → 将文件从 `.work-progress/archive/` 移回 `.work-progress/`。
   - 目录不存在则先创建。

5. **输出摘要** — 告诉用户文件路径、保存了什么、当前状态。

## 更新已有文件的规则

- 勾选执行计划里已完成的子项、推进步骤状态。
- 正在进行的子项描述当前进度（改到哪、卡在哪、半成品在哪些文件）。
- **校验已有认知资产的时效性**：如果代码地图中记录的文件/函数已被重构或删除，修正或删除失效条目，不要只追加新的。
- 补充新增的认知资产、发现、决策。
- 保留历史决策和死路记录。
- 刷新头部"更新"时间。

## 注意事项

- 使用中文撰写内容。
- 只记录有价值的信息，不要流水账。
- **最高优先级三块**：「执行计划」给出推进骨架，「代码认知资产」省下重新探索的成本，「下一步」让接手者立刻能动手。
- 「下一步」必须是动作级语句，包含要改的文件、要跑的命令或要验证的对象。
- 代码地图尽量带 `path:line` / 函数名锚点。
- 验证方式写具体可执行的命令，恢复后能直接跑。
