从 `.work-progress/` 目录恢复工作进度。核心目标：**用进度文件里固化的"代码认知资产"和"执行计划"直接接手任务，不要重新探索代码库**。

## 存储位置

约定为**命令执行所在目录**下的 `.work-progress/`（与 save-progress 一致）。

## 状态管理

状态由目录位置决定：

- `.work-progress/` → 未完成（进行中 / 暂停 / 暂时卡住都放这里）
- `.work-progress/archive/` → 已完成

> 不单独维护额外状态目录。卡住的信息从文件里的「当前卡点 / 风险」读取。

## 执行步骤

1. **扫描目录** — 列出 `.work-progress/` 下的 `.md` 文件（不含子目录），读取每个文件头部信息，按时间戳排序展示：

   ```
   文件名                                    | 状态    | 作用域
   tool-approval-migration_202602121430.md   | 进行中  | services/agent
   persistent-agent_202602131000.md          | 进行中  | 全局
   auth-refactor_202602100900.md             | 卡住    | services/api
   ```

2. **确定要恢复的任务**：
   - **文件名优先**：如果用户通过 $ARGUMENTS 指定了完整文件名，则只恢复该文件。
   - **关键词次优先**：如果用户指定了关键词，则只在匹配该关键词的未完成任务中选择；有唯一明确匹配则恢复。
   - 如果只有一个未完成任务，直接恢复它。
   - 否则列出所有未完成任务供用户选择。
   - 如果没有符合条件的进度文件，直接返回“无可恢复项”，不要猜测，也不要新建文件。

3. **加载上下文** — 读取选中的进度文件，按"恢复优先级"重点吸收：
   - **代码认知资产**：代码地图、项目约定、关键事实。这些是上次会话探索的结晶。
   - **执行计划**：整体推进到哪一步、下一个该攻哪个步骤及其子项。
   - **下一步**：恢复后第一件要做的事，连同前置约束/线索。
   - **关键发现 / 决策**：避免重复推翻已定的方案。
   - **死路 / 坑**：避免重复踩坑。
   - **当前卡点 / 风险**：若任务卡住，先看卡点、解除条件和恢复位置。

4. **按时效性决定校验深度** — 查看进度文件头部的"更新"时间：
   - **1 天以内**：高度信任认知资产，只在"下一步"涉及的文件上做秒级确认（文件还在、关键行号未偏移）。
   - **1-3 天**：信任但校验——对代码地图中的**关键锚点**（入口文件、核心函数）快速 read 确认未被改动。
   - **3 天以上**：认知资产可能已部分失效。对代码地图逐条轻量校验，失效的标注出来提醒用户，但**仍然不要全库重新探索**——未失效的部分继续复用。

5. **输出恢复摘要**：

   ```
   已恢复任务: {标题}
   状态: {进行中/暂停/卡住}  作用域: {作用域}
   上次更新: {更新时间}（{距今 N 天}）

   执行计划进度: 已完成 {N}/{M} 步骤，当前在「{当前步骤}」

   下一步:
   - {动作 + 前置约束}

   已握住的关键认知:
   - {代码认知资产里最关键的 1-3 条}

   当前卡点 / 风险:
   - {没有则写"暂无"；如果卡住，写解除条件和继续位置}

   {若有失效的认知资产:}
   ⚠️ 以下认知可能已过期（上次更新距今 X 天），已做校验:
   - {失效条目及现状}
   ```

6. **进入工作状态** — 恢复完成后，等待用户的下一步指令。不要自动开始执行子项。

## 注意事项

- 使用中文。
- 恢复的第一要务是**复用认知资产**，按时效性决定校验深度，而非一律重新探索。
- 重点关注"代码认知资产""执行计划""下一步""死路"。
- 如果任务状态是「卡住」，优先展示当前卡点和解除条件，让用户判断是否可以继续。
- 找不到明确恢复目标时，不猜测、不新建；列出候选或说明无可恢复项。
- 不要自动开始执行子项，等用户确认后再行动。
- 如果 `.work-progress/` 下没有文件，告知用户当前无未完成任务。
