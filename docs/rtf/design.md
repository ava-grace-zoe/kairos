# rtf 设计文档

## 设计目标

- 目标 1：每个修复都有测试证据支撑——审查发现的问题必须经测试复现确认后才修复
- 目标 2：修复不引入新问题——每次修复后回归验证已有测试
- 目标 3：流程可追踪——每个问题独立循环，状态清晰（fixed / false_positive / fix_failed / regression）
- 非目标：不替代专业安全审计工具、不做性能基准测试、不处理纯架构层面的评审

## 方案概览

- 架构：四阶段串行流程（探查 → 审查 → 测试-修复循环 → 报告），其中测试-修复循环内部逐问题执行
- 关键模块：
  - 探查器：确定审查范围 + 检测测试基础设施
  - 审查器：多维度代码审查，产出结构化问题列表
  - 测试-修复引擎：逐问题执行「测试设计 → 执行 → 修复 → 回归」子循环
  - 报告器：汇总所有问题的最终状态
- 依赖与约束：依赖 git 进行变更获取和回滚；测试执行依赖项目已有的测试框架（无框架时降级为 shell 脚本）

## 关键取舍

### 1. 逐问题循环 vs 批量修复

- 方案 A：逐问题循环——每个问题独立走完测试→修复→回归
- 方案 B：批量修复——一次性修复所有问题，统一回归
- 决策结论：方案 A
- 原因：
  - 可追踪性：每个问题的修复效果独立可验证
  - 可回滚性：单个问题修复失败只回滚该问题，不影响已成功的修复
  - 因果清晰：前一个修复可能改变后续问题的复现条件，逐个处理能自然适应

### 2. 测试先行（TDD 式）vs 直接修复

- 方案 A：先写测试确认问题存在，再修复
- 方案 B：直接修复，事后补测试
- 决策结论：方案 A
- 原因：
  - 测试复现是问题真实性的证据——避免修复不存在的问题（误修）
  - 测试通过是修复有效性的证据——不依赖主观判断
  - 副产物：流程结束后留下可回归的测试用例

### 3. 审查维度：自适应推导 vs 固定检查清单

- 方案 A：根据代码性质自适应推导关注维度
- 方案 B：预设固定的检查清单（如 OWASP Top 10、SOLID 原则等）
- 决策结论：方案 A
- 原因：
  - 固定清单在不同项目/语言间通用性差
  - 自适应推导能根据代码上下文聚焦最相关的维度
  - 避免「清单驱动」导致的形式化审查（逐项打勾但忽略真正问题）

### 4. 误报处理：宁多勿漏 vs 精确审查

- 方案 A：审查阶段宽松报告，由测试阶段过滤误报
- 方案 B：审查阶段严格过滤，只报告高置信度问题
- 决策结论：方案 A
- 原因：
  - 审查阶段的误报成本低（只多写一个测试）
  - 漏报成本高（真实问题未被发现和修复）
  - 测试复现是天然的误报过滤器

## 数据与接口

- 输入：
  - 用户指定的审查对象（diff / 文件路径 / 目录 / PR URL）
  - 项目代码库（git 仓库）
- 输出：
  - 综合报告（对话中直接输出）
  - 修复后的代码变更（直接修改文件）
  - 新增的测试文件（留在项目中供后续回归）
- 中间状态：
  - `reviewScope`：审查范围（文件列表 + 行范围）
  - `testInfra`：测试基础设施信息
  - 问题列表：每个问题的结构化描述和状态流转
- 兼容性说明：
  - 语言无关——通过探查阶段适配不同语言的测试框架
  - 无测试框架时降级为 shell 脚本验证

## 风险与缓解

### 1. 测试设计质量不足导致误报率高

- 触发条件：问题描述模糊或 `reproduction_hint` 不够具体
- 影响：大量问题被标记为 `false_positive`，真实问题可能被漏过
- 缓解措施：`reproduction_hint` 要求包含具体的输入数据和预期行为；测试执行错误与测试通过区分处理

### 2. 修复引入隐蔽回归

- 触发条件：项目已有测试覆盖率低，回归验证无法捕获所有影响
- 影响：修复通过回归但实际引入了新问题
- 缓解措施：最小化修复原则（只改必要的代码）；修复后的 diff 在报告中完整展示供用户复核

### 3. 逐问题循环耗时过长

- 触发条件：审查发现大量 critical/warning 问题
- 影响：用户等待时间过长
- 缓解措施：按 severity 排序优先处理 critical；单个问题修复重试上限 2 次；用户可随时中断
