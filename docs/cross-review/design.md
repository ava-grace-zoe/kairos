# cross-review 设计方案

## 设计目标

1. 最小化跨 agent 传递的 prompt 长度（≤30 字）
2. 利用文件系统作为 agent 间的信息传递通道
3. 评审产物集中存放在系统临时目录，方便审计
4. 支持并行调用多个外部 agent

## 方案概览

```
用户触发 cross-review
        │
        ▼
  ┌─ prepare ─────────────────────────┐
  │  方案在文件系统？                    │
  │  ├─ 是 → 记录绝对路径               │
  │  └─ 否 → 写入 $TMPDIR/.cross-review/.../context.md│
  │  生成 instructions.md                              │
  └───────────────────────────────────┘
        │
        ▼
  ┌─ dispatch ────────────────────────┐
  │  并行调用：                        │
  │  ├─ claude -p "阅读 {path} 并执行" │
  │  └─ codex exec "阅读 {path} 并执行"│
  │  各 agent 读指令文件 → 写结果文件    │
  └───────────────────────────────────┘
        │
        ▼
  ┌─ synthesize ──────────────────────┐
  │  读取 review-*.md                  │
  │  交叉对比 → 共识/分歧/独特洞察      │
  │  生成 synthesis.md + 输出摘要       │
  └───────────────────────────────────┘
```

## 关键取舍

### 1. 文件系统传递 vs prompt 内联

**选择：文件系统传递**

- prompt 内联方案内容会导致 prompt 极长，且不同 agent 对长 prompt 的处理能力不同
- 文件系统传递只需在 prompt 中写路径（~30 字），agent 自行读取文件获取完整上下文
- 额外好处：评审指令文件可复用，方便调试和回溯

### 2. 临时目录位置：$TMPDIR/.cross-review/

**选择：`$TMPDIR/.cross-review/`**

- 所有评审产物集中在 `$TMPDIR/.cross-review/` 下，按时间戳分目录，方便临时审计
- 不污染项目目录和 git 状态
- 系统重启后自动清理
- 每次评审使用时间戳命名子目录（`<YYYYMMDD-HHMMSS>`），天然隔离

### 3. 方案内容拷贝 vs 路径引用

**选择：路径引用优先**

- 方案已在文件系统时直接引用绝对路径，零拷贝
- 仅对话中的方案内容才写入临时文件
- 减少不必要的 IO 和重复数据

### 4. 评审指令传递方式：prompt 参数 vs 指令文件

**选择：指令文件（instructions.md）**

- 将评审维度、输出格式、文件路径统一写入指令文件
- prompt 只说"阅读指令文件并执行"
- agent 读文件时自然看到所有要求，无需 prompt 编码复杂结构
- 指令文件可独立检查和调试

## Agent 分发规则

### 可用 Agent 列表

| Agent | CLI 命令 | 模型架构 | 关键参数 |
|-------|----------|---------|---------|
| Claude | `claude -p "{prompt}"` | Claude 系列 | `--output-format text` |
| Codex | `codex exec "{prompt}"` | GPT 系列 | `--full-auto` |

### 分发逻辑

1. **自我识别** — 根据当前运行环境判断自身身份（如 Claude Code 内执行 → 自身为 Claude）
2. **排除自身** — 从可用 agent 列表中排除与自身同架构的 agent
3. **可用性检测** — 对剩余 agent 检测 CLI 是否存在（`which claude` / `which codex`）
4. **分发执行** — 向所有可用的外部 agent 并行发送评审任务

### 兜底：无法确定自身身份

当执行环境无法明确判断自身所属模型架构时，向用户确认发送目标：
- 列出所有检测到可用 CLI 的 agent
- 由用户选择发送给哪些 agent

### 退化场景

- 仅一个外部 agent 可用 → 单 agent 评审，synthesize 阶段跳过交叉对比，直接输出该 agent 结果
- 无外部 agent 可用 → 提示用户安装至少一个外部 agent CLI，终止流程

## 数据与接口

### instructions.md 结构

```markdown
# 评审任务

## 待评审文件
- {绝对路径}

## 背景
{方案要解决的问题}

## 关注点
{用户指定或默认}

## 评审维度
1. 正确性
2. 可维护性
3. 安全性
4. 性能
5. 遗漏

## 输出格式（严格遵守）
### [维度名]
- **问题**: ...
- **严重程度**: high/medium/low
- **建议**: ...

### 总体评价
{1-2 句话}
```

### prompt 模板

```
阅读 {tmpdir}/instructions.md 中的评审任务并执行。将评审结果写入 {tmpdir}/review-{agent}.md
```

### synthesis.md 结构

```markdown
# 交叉评审综合报告

## 参与评审的 Agent
- claude / codex

## 共识问题（高优先级）
多个 agent 共同指出的问题

## 分歧点（需人工判断）
仅单个 agent 提出的问题

## 独特洞察
值得关注的非共识发现

## 行动建议
按优先级排序的改进项
```

## 风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| 外部 agent CLI 未安装 | 中 | dispatch 前检测，不可用则跳过并提示 |
| agent 无文件读写权限 | 中 | codex 需 `--full-auto`；claude -p 默认有读权限 |
| agent 不遵守输出格式 | 低 | synthesize 阶段做宽松解析，不强依赖格式 |
| 临时目录被提前清理 | 极低 | 单次会话内完成全流程，系统重启前可审计 |
