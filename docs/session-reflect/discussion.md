# session-reflect 讨论记录

## 决策记录

### 2026-03-01 维度体系重构

- 主题：评分维度从 5 个缩减为 3 个，后进一步演化为 5 类证据直接驱动
- 背景：对原 5 维度方案进行审查，发现可观测性、稳定性、评分适用性三方面存在硬伤。
- 演化路径：5 维度评分 → 3 维度评分 → 移除评分机制，改为 5 类证据（沟通偏好、表达模式、协作偏好、工程偏好、决策与风险偏好）直接驱动画像生成。
- 结论：评分是不必要的中间环节，证据直接转化为行为指令更可靠。
- 影响范围：SKILL.md、design.md、index.md

### 2026-03-01 变更历史独立文件

- 主题：评价历史 + 版本记录合并后从 context.md 移出
- 背景：context.md 通过软链接被 AI 读取，应确保每行都是 AI 需要消费的信息。
- 结论：合并为 `~/.agents/history.md`，追加写入，context.md 中不保留任何审计信息或路径指针。
- 影响范围：SKILL.md、template.md

### 2026-03-01 context.md 去噪

- 主题：移除 context.md 中 AI 不需要消费的内容
- 背景：首次端到端测试后发现 context.md 充斥评分数字、可观测信号、元数据、HTML 注释等执行细节，对 AI 来说是噪音。
- 结论：
  - context.md 只保留 4 个章节：基本信息、综合画像、AI 应对指南、偏好与习惯。
  - 移除：维度评价章节、元数据章节、排除项章节、所有 HTML 注释。
  - 评分是生成 AI 应对指南的中间过程，分数只进 history.md。
  - AI 应对指南中不出现分数，只写行为描述。
- 影响范围：SKILL.md、template.md、context.md

### 2026-03-01 template.md 删除

- 主题：删除独立模板文件，改为内联
- 背景：精简后模板只剩几个标题和"暂无数据"，单独维护没有意义。
- 结论：初始结构内联到 SKILL.md（章节校验时的补齐依据）和 setup.sh（heredoc 创建），删除 `assets/template.md` 和 `assets/` 目录。
- 影响范围：SKILL.md、setup.sh、目录结构

### 2026-03-02 双轨重设计：偏好 + 认知

- 主题：从纯偏好同步升级为「偏好校准 + 元认知提升」双轨体系
- 背景：审查现有画像发现两个结构性问题：(1) 5 类证据全部是偏好维度，缺失认知盲区、决策偏误等关键维度，导致画像全部是正面/中性描述；(2) AI 应对指南逻辑是"用户永远正确，AI 适配"，缺少 AI 主动挑战用户决策的防护机制。
- 演化路径：5 类偏好证据 → A 轨（偏好 A1-A5）+ B 轨（认知 B1-B5）双轨结构
- 关键设计决策：
  - 引入认知心理学框架（System 1/2、Dunning-Kruger、Cognitive Biases、Growth Mindset、Bias Blind Spot）作为 B 轨提取框架
  - context.md 综合画像拆为「协作风格」（A 轨驱动）+「思维模式」（B 轨驱动）
  - AI 应对指南新增「认知防护」子章节（B 轨驱动），AI 在特定场景下主动提醒/挑战用户
  - B 轨元认知洞察直接在会话中输出给用户，不写入 context.md
  - history.md 格式适配双轨（A 轨证据 + B 轨证据 + 元认知反馈摘要）
- 结论：双轨互补——A 轨让 AI 更好配合用户，B 轨让 AI 在关键时刻主动挑战用户，同时帮助用户自我觉察和成长。
- 影响范围：SKILL.md、design.md、index.md、test-plan.md、setup.sh

### 2026-03-03 A 轨维度扩展与硬性禁区改造

- 主题：A 轨从 5 类扩展为 6 类，子维度细化，硬性禁区改为成对记录
- 背景：对照外部"深度协作偏好与元信息提取"提示词，发现三个结构性缺口——A2 表达模式描述过泛、A4 缺少工程权衡价值观、完全缺失"问题解决工作流"维度；硬性禁区只记录"不要做什么"，缺少正面引导。
- 变更：
  - A2 增加 XY 问题倾向、上下文遗漏类型（环境/版本/依赖/日志截断）、抽象度偏差
  - A4 增加工程权衡默认立场（糙快猛 vs 防御性编程）、反复踩的坑和验证有效的模式
  - 新增 A6 问题解决工作流：自顶向下 vs 自底向上、遇到报错时要答案还是要排查思路、架构设计 vs 实现细节的优先级倾向
  - 协作风格增加问题解决工作流和工程权衡立场
  - 硬性禁区改为成对记录：每条禁区附带"替代做法"
- 结论：这些维度是跨项目稳定的用户级特征，直接影响 AI 配合用户的节奏和方式。成对记录让 AI 同时知道禁区和替代方案，可执行性更强。
- 影响范围：SKILL.md、design.md

### 2026-05-28 性格与思维轮廓持久化及工作流 Skill 化升级

- 主题：B 轨全面持久化性格缺陷画像，A 轨新增 A7 高频重复工作流识别与 Skill 固化建议
- 背景：在审查 B 轨时发现，原先将“认知洞察”作为一次性会话反馈丢弃的设计，导致后续交互的 AI 依然无法获知并消费用户的认知盲区，防护大打折扣。同时，用户在会话中经常会陷入手动执行重复步骤的“工程搬砖”怪圈。
- 结论：
  - **B 轨重构**：升级为 `### 性格与思维轮廓`（持久化写入性格形状与缺陷，带具体行为证据）和 `### 性格与认知防护`（针对缺陷的 AI 主动防御指令），不回避性格标签，追求写实。
  - **A 轨新增 A7 轨**：识别整场会话中出现 2 次及以上的高频重复行为流程，在会话输出中以 `💡 工程效能建议` 的形式，主动提示用户将其固化落地为标准的 Agent Skill（联动 `skill-creator`）。
- 影响范围：SKILL.md、setup.sh、index.md、design.md、discussion.md

### 2026-06-30 v3 合约收敛与写回确认门禁

- 主题：统一 `session-reflect` 的文档、执行合约和初始化模板，并增加修改前确认机制。
- 背景：`changelog.md` 已声明 v3 使用 `### 性格与思维轮廓` 和 `### 性格与认知防护`，但 `SKILL.md` 与 `setup.sh` 仍保留旧章节 `### 思维模式` / `### 认知防护`；同时 A7 高频重复工作流只存在于文档，没有进入执行合约。
- 结论：
  - 当时的单文件 `context.md` 标准章节统一为：`协作风格`、`性格与思维轮廓`、`行为校准`、`性格与认知防护`、`硬性禁区`。
  - A7 正式纳入 A 轨，但只输出工程效能建议并写入 `history.md`，不默认写入 `context.md`。
  - 新增强制确认门禁：创建、覆盖、软链接或追加任何文件前，必须先展示拟修改摘要并获得用户明确确认。
  - B 轨负面或缺陷类观察必须满足证据门槛；证据不足时标为“初步观察”，默认不进入稳定画像。
  - 注：该单文件章节方案已被后续三层拆分和目录化模型取代。
- 影响范围：SKILL.md、setup.sh、index.md、design.md、test-plan.md、changelog.md

### 2026-06-30 context.md 写作人称规范

- 主题：统一 `context.md` 的人称，降低跨工具读取时的指令歧义。
- 背景：现有画像中混用“用户”“AI 应”“我”等表达，`context.md` 又会被 Claude / Codex / Gemini 直接读入上下文；第一人称容易被模型误解为助手自身，第三人称指令又不如第二人称可执行。
- 结论：
  - `综合画像` 使用第三人称“使用者”。
  - `AI 应对指南` 使用第二人称命令式“你”。
  - `偏好与习惯` 使用名词化短句。
  - 禁止用第一人称“我”表达用户偏好。
  - 注：该人称规范后续收敛为 `profile.md` 使用第三人称、`context.md` 使用第二人称。
- 影响范围：design.md、index.md、test-plan.md、SKILL.md

### 2026-06-30 三层文件模型：profile / context / history

- 主题：将画像沉淀与 Agent 常驻执行策略分离。
- 背景：继续讨论后确认，用户画像本身不应直接丢给日常 Agent。画像是分析中间层，直接常驻会带来心理诊断感、标签误用和上下文污染；Agent 更需要的是从画像反推出的可执行策略。
- 结论：
  - 新增 `~/.agents/profile.md`：保存跨会话稳定画像、思维模式、风险线索和证据索引，供 `session-reflect` 更新时参考。
  - `~/.agents/context.md` 调整为 Agent 执行策略层，只写“你应该怎么做”，不写“使用者是什么样的人”。
  - `~/.agents/history.md` 继续作为审计日志，记录每次更新的证据、推导和变更摘要。
  - 更新流程变为：会话证据 → 更新 profile 草案 → 从 profile 反推 context 策略草案 → 用户确认 → 写回三层文件。
  - 注：该路径方案已被后续“候选工作流库与目录化组织”收纳，当前实现路径以 `~/.agents/session-reflect/` 为准。
- 影响范围：design.md、index.md、test-plan.md、changelog.md；后续需要同步 SKILL.md 与 setup.sh

### 2026-06-30 候选工作流库与目录化组织

- 主题：为跨会话重复工作流增加独立候选库，并收纳 session-reflect 相关文件。
- 背景：A7 原本只能识别当前会话里的重复工作流，容易过早推荐创建 skill。用户指出需要能沉淀“不同会话多次出现的重复”，达到阈值后再推荐。
- 结论：
  - 新增 `workflow-candidates.md`，记录候选工作流的状态、出现次数、最近出现、输入、输出、证据和推荐动作。
  - 候选规则：单次会话重复记为 `candidate`；跨会话 2 次可升级 `recommended`；跨会话 3 次且输入输出稳定时强建议 skill 化。
  - 内部文件收纳到 `~/.agents/session-reflect/`，包括 `profile.md`、`context.md`、`history.md`、`workflow-candidates.md`。
  - 保留 `~/.agents/context.md` 作为对外稳定入口，推荐指向 `session-reflect/context.md`。
- 影响范围：design.md、index.md、test-plan.md、changelog.md；后续需要同步 SKILL.md 与 setup.sh
